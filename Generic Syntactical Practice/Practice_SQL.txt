CREATE TABLE customers (
  id SERIAL PRIMARY KEY,
  first_name VARCHAR(50),
  last_name VARCHAR(50),
  email VARCHAR(100),
  city VARCHAR(50),
  country VARCHAR(50),
  signup_date DATE
);

CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100),
  category VARCHAR(50),
  price NUMERIC(10,2),
  stock_quantity INT
);

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  customer_id INT REFERENCES customers(id),
  product_id INT REFERENCES products(id),
  quantity INT,
  order_date TIMESTAMP DEFAULT NOW(),
  status VARCHAR(20)
);

SELECT * FROM orders ORDER BY order_date DESC LIMIT 10;
FETCH Alternative
SELECT * FROM orders ORDER BY order_date DESC ROWS FETCH FIRST 10 ROWS ONLY;

SELECT * FROM orders ORDER BY order_date DESC LIMIT 10 OFFSET 10;

The core difference is that LIMIT/OFFSET is a platform-specific dialect (used in MySQL and PostgreSQL), while FETCH is the official ANSI SQL standard. Mechanically, they execute the exact same way inside the database engine. The reason both paginate poorly at scale is due to how OFFSET works under the hood. The Problem: O(N) Sequential Scanning.


On an unindexed column, both operators force a full table scan, checking every single row manually. Because the database must read all data from the disk anyway, the minor CPU overhead of ILIKE performing case-insensitive character comparisons is negligible. Both scale poorly.


The WHERE clause operates strictly on raw rows before any grouping or aggregation takes place. It acts as an early row-level gatekeeper, discarding unmatched data so the database avoids wasting CPU and memory resources grouping irrelevant records. Aggregated groups are filtered later using HAVING.
